## 2. Methodology: loglikelihood MC on instruct/thinking models

- **Accepted practice: yes, with caveats.** Raw-completion loglikelihood MC is the historical lm-eval default and was applied uniformly to base and instruct models in Open LLM Leaderboard v1. **OLL v2 switched to recommending `--apply_chat_template` (+ `--fewshot_as_multiturn`) for instruction models** ([HF leaderboard docs](https://github.com/huggingface/leaderboards/blob/main/docs/source/en/open_llm_leaderboard/about.md)). lm-eval supports chat template *with* loglikelihood on completions-style backends (hf, vllm, local-completions) — chat-completions APIs can't do loglikelihood at all ([API guide](https://github.com/EleutherAI/lm-evaluation-harness/blob/main/docs/API_guide.md)).
- **Bias direction:** on *generative* tasks the template strongly favors instruct models (e.g., large IFEval uplifts — [TDS experiment](https://towardsdatascience.com/evaluations-with-chat-formats-7604067023c9/), Zephyr +39%). On *loglikelihood MC* the effect is smaller and task/model-dependent; the usual net direction is that no-template mildly **underestimates** instruct models, but instruct models remain robust MC selectors without templates ([arXiv:2404.08382](https://arxiv.org/html/2404.08382v1)); chat models' raw probabilities are miscalibrated yet still rank answers usefully ([arXiv:2402.13213](https://arxiv.org/html/2402.13213)). **For your study the defensible setup is: identical raw-completion prompts for every model (perfectly comparable, slight absolute underestimate for instruct models), optionally a second chat-templated pass as sensitivity analysis.** Report which regime you used; don't mix.
- **SEA-HELM does the opposite:** generation-based scoring with chat templates always applied (instruct = model's own template; `--base_model` loads `chat_templates/base_model.jinja` and disables MT-Bench), 8 runs/model ([SEA-HELM GitHub](https://github.com/aisingapore/SEA-HELM)). So your loglikelihood numbers will **not** be comparable to SEA-HELM leaderboard numbers — cite them as external context only.
- **Thinking mode: confirmed irrelevant/should be off.** Loglikelihood scores fixed continuations; nothing is generated, so no `<think>` block can occur. lm-eval **v0.4.12 explicitly disallows `enable_thinking` with `multiple_choice`/`loglikelihood` tasks** ([v0.4.12 release notes, Zenodo](https://zenodo.org/records/20122284)). If you apply chat templates, pass `enable_thinking=False` in chat-template kwargs so the template doesn't inject an empty think-prefix; with no chat template the question doesn't arise. Known generative-mode extraction bugs with thinking ([#3419](https://github.com/EleutherAI/lm-evaluation-harness/issues/3419), [#3576](https://github.com/EleutherAI/lm-evaluation-harness/issues/3576)) don't affect loglikelihood tasks.

## 3. lm-eval 0.4.12 × llama.cpp

- **Adapter:** `--model gguf` (alias `ggml`), `--model_args base_url=http://localhost:8080` (+optional `max_length`). It POSTs to `{base_url}/v1/completions` with `logprobs=10, echo=True, temperature=0, max_tokens=1` and computes loglikelihood from `logprobs.token_logprobs`/`text_offset`/`tokens` — i.e., it **requires OpenAI-style prompt-token logprobs via echo** ([gguf.py](https://github.com/EleutherAI/lm-evaluation-harness/blob/main/lm_eval/models/gguf.py), [original PR #617](https://github.com/EleutherAI/lm-evaluation-harness/pull/617)).
- **It was written for `llama-cpp-python`'s server, NOT ggml-org `llama-server`.** Against `llama-server` it fails with `AssertionError: Invalid response for loglikelihood` because llama-server returns no prompt logprobs — **open issue [#2525](https://github.com/EleutherAI/lm-evaluation-harness/issues/2525)** (also older [#1437](https://github.com/EleutherAI/lm-evaluation-harness/issues/1437)).
- **llama-server status:** the [server README](https://github.com/ggml-org/llama.cpp/blob/master/tools/server/README.md) documents no `echo` parameter on `/v1/completions`; the native `/completion` endpoint's `n_probs`/`post_sampling_probs` cover **generated** tokens only. I found **no evidence that 2026 builds added echo+prompt-logprobs** (absence flagged as unverified — grep a current build's `server` code to be certain). Therefore `--model local-completions` against llama-server also cannot do loglikelihood; `local-completions` + echo works with **vLLM's** OpenAI server (modulo old echo quirks, [vllm#5334](https://github.com/vllm-project/vllm/issues/5334)).
- **Practical options for D/E:**
  1. **`llama-cpp-python[server]`** (supports `echo`+`logprobs`) + `--model gguf`. Caveats: needs a CUDA aarch64 source build (`CMAKE_ARGS="-DGGML_CUDA=on -DCMAKE_CUDA_ARCHITECTURES=121a"` pip install), single-request no batching → slow for Belebele (900×4 requests), historical logprob-accuracy bug [#349](https://github.com/abetlen/llama-cpp-python/issues/349) (old, but sanity-check a few items against hf backend), and its bundled llama.cpp must be new enough for `qwen3moe` (fine) and Qwen3.6 arch for model D (check).
  2. **lm-eval `hf` backend with `gguf_file=`** (transformers dequantizes GGUF to fp32/bf16): exact, batched, but slow to load and memory-hungry; on a 128 GB Spark it is feasible for the 18.6 GB Q4_K_M and is the most trustworthy fallback ([lm-eval README](https://github.com/EleutherAI/lm-evaluation-harness/blob/main/README.md)). Note this measures the *quantized weights* under a different (dequantized) matmul path than llama.cpp's kernels.
  3. Patch `gguf.py` to call llama-server's native `/completion` with tokenized continuations — nonstandard, not recommended for a paper.

## 4. `ghcr.io/ggml-org/llama.cpp:server-cuda` on DGX Spark (GB10, aarch64)

- **Multi-arch: yes, now.** Current `server-cuda` (e.g., `server-cuda12-b9603`, ~June 2026) manifests list **linux/amd64 + linux/arm64**; `server-cuda13` likewise ([GHCR](https://github.com/orgs/ggml-org/packages/container/llama.cpp/938474228?tag=server-cuda), [docker.md](https://github.com/ggml-org/llama.cpp/blob/master/docs/docker.md)). Historical context: arm64-CUDA was absent and even closed "not planned" in Nov 2025 ([#17321](https://github.com/ggml-org/llama.cpp/issues/17321)); community images exist specifically for Spark ([ardge-labs/llama-cpp-dgx-spark](https://github.com/ardge-labs/llama-cpp-dgx-spark), sm_121/CUDA 13; [NVIDIA forum thread](https://forums.developer.nvidia.com/t/building-llama-cpp-container-images-for-spark-gb10/353664)).
- **⚠ GB10 caveat (partially unverified):** the `server-cuda` Dockerfile defaults to CUDA **12.8.1** with `CUDA_DOCKER_ARCH=default` ([cuda.Dockerfile](https://github.com/ggml-org/llama.cpp/blob/master/.devops/cuda.Dockerfile)); CUDA 12.8 cannot emit sm_121 SASS (GB10 = compute 12.1, needs CUDA ≥12.9/13). It may run via PTX JIT, but **prefer `server-cuda13`** or build from source the way NVIDIA's own playbook does: `cmake -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES=121a-real` ([dgx-spark-playbooks/nvidia/llama-cpp](https://github.com/NVIDIA/dgx-spark-playbooks/tree/main/nvidia/llama-cpp), [build.nvidia.com/spark/llama-cpp](https://build.nvidia.com/spark/llama-cpp)). llama.cpp is officially benchmarked on Spark ([benches/dgx-spark](https://github.com/ggml-org/llama.cpp/blob/master/benches/dgx-spark/dgx-spark.md)). Docker GPU images are "not tested by CI beyond being built".
- **Flags:** `-ngl 99` (offload all layers; unified memory makes `--cpu-moe`/`-ncmoe` unnecessary), `-fa` is now `on|off|auto` with default `auto` (explicit `-fa on` is fine), `-c <ctx>`; add `--host 0.0.0.0 --port ...`. NVIDIA's Spark example serves `unsloth/Qwen3.6-35B-A3B-MTP-GGUF:UD-Q4_K_XL` with optional MTP speculative decoding (`--spec-type draft-mtp`).
- **qwen3moe GGUF: yes** — supported since **b5092** (Apr 2025) ([Qwen docs](https://qwen.readthedocs.io/en/latest/run_locally/llama.cpp.html)); Typhoon2.5-30B-A3B's GGUF is `qwen3moe` and Spark tutorials run Qwen3-235B MoE ([forum](https://forums.developer.nvidia.com/t/tutorial-build-llama-cpp-from-source-and-run-qwen3-235b/352604)). Model D's **Qwen3.6 dense-hybrid arch needs a much newer build** (Qwen3.6 support landed ~May 2026) — use one recent image/build for both D and E.
